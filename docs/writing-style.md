# ADR 성격 기술 글쓰기 가이드

이 블로그는 문제 해결 과정과 기술적 선택의 근거를 기록하는 데 초점을 둡니다.
고정 템플릿으로 강제하지 않고, 일반적인 기술 블로그 흐름을 따르되 아래 순서를 권장합니다.

## 권장 글 흐름

1. 문제 상황
   - 무엇이 문제였는지, 어떤 맥락에서 발생했는지 설명합니다.
2. 시도/탐색
   - 어떤 가설을 세웠고 무엇을 확인했는지 기록합니다.
3. 최종 선택
   - 실제로 채택한 해결책을 명확히 제시합니다.
4. 선택 근거
   - 왜 이 선택을 했는지, 어떤 트레이드오프를 감수했는지 적습니다.
5. 대안과 기각 이유
   - 검토한 대안과 채택하지 않은 이유를 간단히 남깁니다.
6. 결과/회고
   - 적용 결과, 배운 점, 다음 개선 아이디어를 정리합니다.

## 포스트 네이밍 규칙

- 파일명: `YYYY-MM-DD-adr-<slug>.md`
- 제목 권장: `ADR: <핵심 의사결정>`

## 태그 권장

- `adr`
- `decision`
- `troubleshooting`

## 작성 원칙

- 짧고 명확하게 씁니다.
- 결론보다 근거를 우선합니다.
- 실패한 시도도 필요한 범위에서 기록합니다.
